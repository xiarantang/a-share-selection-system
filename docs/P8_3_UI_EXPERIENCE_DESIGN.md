## 1. 当前 UI 问题诊断

基于 `app.py`（P8.2-4 后）逐区审查，发现以下具体问题：

### 1.1 首页：小白第一眼是否知道怎么用

**现状**（`app.py:277-308`）：

- 顶部标题 + 副标题 + 就绪状态横幅 + 首次使用检查 expander + 四步引导卡片。
- 就绪横幅有两行重复语义：`st.success("可以开始选股")` 紧跟 `st.info("当前主要使用 A 股备用数据通道")`。
- "首次使用检查" expander 在首次未选股时 `expanded=True`，占据大量屏幕空间，小白需要滚过它才能看到四步引导。
- 四步引导卡片本身清晰，但排列为四列，每列文字密集；移动端或窄屏时挤压严重。

**问题**：

| 编号 | 问题 | 严重度 |
|------|------|--------|
| H-1 | 就绪横幅信息重复（success + info 两行说类似的事） | 中 |
| H-2 | "首次使用检查" expander 默认展开，把四步引导推到屏外 | 高 |
| H-3 | 四步引导文字偏密，关键行动点（点左侧按钮）不够突出 | 中 |
| H-4 | 左侧栏「选股参数」「一键运行」「查看结果」三段割裂，小白不确定先看哪里 | 中 |

### 1.2 结果页：是否第一眼看到 Top 3、整体质量、主要风险

**现状**（`app.py:470-529`）：

- 选股完成后，Tab 外先展示 4 列信息条（股票池/起始日/扫描数/数据源），然后 5 列覆盖摘要（数据范围/最少K线/最多K线/平均K线/覆盖不全），再一行纯文本覆盖摘要，再验证摘要卡片（整体质量/平均评分/覆盖不足率/低置信度/高风险），最后 4 个 Tab。
- Top 候选藏在 `tab1 (候选表格)` 里，必须点击 Tab 才能看见。
- 没有 "Top 3 速览"——小白看到的第一屏全是数据源统计和覆盖摘要，不是"哪几只最好"。

**问题**：

| 编号 | 问题 | 严重度 |
|------|------|--------|
| R-1 | Top 候选必须点 Tab 才能看到，首屏没有候选速览 | **高** |
| R-2 | 首屏信息密度高但信息价值低（4 列信息条 + 5 列覆盖 + 1 行纯文本，三层说同一件事） | 高 |
| R-3 | 验证摘要和数据覆盖摘要分开显示，小白要拼凑"整体好不好" | 中 |
| R-4 | 无 Top 3 卡片式速览，无法一眼看出谁排前面、为什么、风险如何 | **高** |

### 1.3 逐只详情：是否解释清楚但不吓人

**现状**（`app.py:566-621`）：

- P8.2-3 已在 expander 顶部展示 explain.summary，加分/风险分两列，可靠性折叠。
- 因子得分展示为纯文本 `趋势25/动量18/量能12/风控16/数据质量8/形态7`，无视觉对比。
- 因子数值展示为 `MA5:xx MA20:xx MA60:xx | 20日动量:xx% | RSI:xx | 波动率:xx%`，术语密集。
- reasons 和 risks 分别用 `st.success` 和 `st.error` 包裹，颜色与风险等级无对应。

**问题**：

| 编号 | 问题 | 严重度 |
|------|------|--------|
| D-1 | 因子得分无视觉对比，无法一眼看出强项弱项 | 中 |
| D-2 | 因子数值术语密集（MA5/MA20/MA60/RSI/波动率），小白看不懂 | 低 |
| D-3 | 风险等级（low/medium/high）无颜色区分，全靠文字 | 中 |
| D-4 | `st.error` 展示 risks 内容，即使风险很低也显示红底，可能吓到小白 | 中 |

### 1.4 整体风格：哪些地方仍然像工程工具

| 编号 | 问题 | 位置 |
|------|------|------|
| E-1 | 结果页信息布局以"工程调试"逻辑排列：数据源→覆盖→验证→Tab，而非"产品"逻辑：Top3→风险→数据→详情 | 结果页整体 |
| E-2 | 左侧栏用 `st.slider` / `st.selectbox` 等原生控件，无分组标题视觉层次 | 侧栏 |
| E-3 | 选股等待时 spinner 文字"正在拉取数据、计算评分"是工程阶段描述，无进度感知 | `app.py:420` |
| E-4 | 决策标签 `strong_watch` / `watch` / `avoid` 为英文内部术语，候选表格中直接显示 | `app.py:552` |
| E-5 | "数据区间" `2025-11-13~2026-05-15` 格式偏技术，小白更习惯"近 6 个月" | 候选表格 |

## 2. P8.3 总目标

让小白打开页面后，不需要懂命令行，也能完成以下流程：

```
启动 → 开始选股 → 看 Top 3 → 看原因 → 看风险 → 看数据质量 → 看报告
```

具体标准：

1. **首页**：小白打开后 3 秒内知道"点左侧按钮开始"，不需要滚动或猜。
2. **等待**：选股运行时，小白知道当前在做什么、大概还要多久。
3. **结果首屏**：不需要切 Tab，第一眼看到 Top 3 候选、整体质量、主要风险。
4. **逐只详情**：点开某只股票后，先看到一句话解释，再看加分/扣分，再看数据质量；因子用中文标签，风险用颜色区分。
5. **报告**：仍在 Tab 内，内容不变，生成方式不变。

### P8.3-1：首页/启动区体验优化

**目标**：小白打开页面 3 秒内知道下一步该做什么，不被信息淹没。

**允许修改的文件**：
- `app.py`（仅首页区域：`render_step_guide()` 函数、就绪横幅、首次使用检查 expander）

**禁止修改的文件**：
- `strategies/selection.py`
- `data/fetcher.py`
- `reports/generator.py`
- `validation/` 目录
- `main.py`

**具体改动**：

| 改动 | 说明 | 解决的问题 |
|------|------|-----------|
| 合并就绪横幅 | 将 success + info 两行合并为一行，简洁明了 | H-1 |
| 首次使用检查默认折叠 | 改为 `expanded=False`，不占据首屏 | H-2 |
| 简化四步引导为两步 | 合并为"① 在左侧选参数 → ② 点开始选股"，四步保留为折叠细节 | H-3 |
| 侧栏简化分组 | 将「选股参数」「一键运行」「查看结果」三段合并为更清晰的视觉分组 | H-4 |

- "首次使用检查"默认收起，点击可展开。

## 4. UI 设计原则

### 4.1 必须坚持

| 编号 | 原则 | 说明 |
|------|------|------|
| P-1 | 不做营销落地页 | 不加动画、Banner、滚动特效、营销文案。功能优先。 |
| P-2 | 不做复杂前后端分离 | 继续用 Streamlit，不引入 React/Vue/Angular。 |
| P-3 | 不引入登录/数据库 | 无用户管理、无持久化存储。 |
| P-4 | 不引入 AI/qlib | AI 不参与评分、排序、解释生成。P8.2 的 explain 是规则生成，不是 AI。 |
| P-5 | 不输出买入/卖出/目标价/收益预测 | 解释只说"为什么排前面"，不说"该不该买"。 |
| P-6 | 保持双击 `start_ui.command` 路径 | 小白启动方式不变。 |
| P-7 | 保持 CLI 验收命令不变 | `py_compile`、`select`、`backtest-validate`、`report` 必须始终通过。 |
| P-8 | 不改评分公式和因子权重 | UI 只改展示，不改底层逻辑。 |
| P-9 | 不改数据链路 | 不改 akshare/baostock/skill_fallback 优先级和降级逻辑。 |
| P-10 | 每步必须向后兼容 | JSON 字段不变，新增字段不破坏旧字段。 |

### 4.2 设计偏好

| 偏好 | 说明 |
|------|------|
| 信息层次分明 | 重要信息（Top 3、整体质量、风险）在首屏；次要信息（数据源、覆盖细节）可折叠。 |
| 中文优先 | 小白看到的标签、按钮、提示全部中文；技术术语（MA5/RSI）默认折叠。 |
| 颜色有意义 | 红色=高风险、橙色=中风险、绿色=低风险，全页一致。 |
| 少即是多 | 同一信息不在页面上出现两次。信息条、覆盖摘要、验证摘要合并。 |
| 渐进式展示 | 首屏给结论和速览；点开看详情；再点开看技术指标。 |

## 5. 下一步推荐

**推荐：只先执行 P8.3-1（首页/启动区体验优化）。**

理由：

1. P8.3-1 改动最小，只涉及 `render_step_guide()` 和就绪横幅，风险最低。
2. P8.3-1 不涉及结果展示区域，不影响选股、验证、报告。
3. P8.3-1 完成后可以截图验收，确认首页改善效果。
4. P8.3-2（Top 3 卡片）是最大的单步改动，应该独立验收后再做。
5. P8.3-3（风险视觉）和 P8.3-2 有依赖关系（风险颜色需要在 Top 3 卡片中使用），但各自独立提交。
6. P8.3-4（验收脚本）依赖前三步全部完成。

**执行顺序**：P8.3-1 → P8.3-2 → P8.3-3 → P8.3-4

**每步完成后**：
- 更新 `PROJECT_STATE.md`
- `python3 -m py_compile app.py` 通过
- `python3 main.py select --universe static --limit 10 --top 5` 通过
- `python3 main.py backtest-validate` 通过
- `python3 main.py report` 通过
- `git commit` + `git push`
