# P8 阶段路线图

> 规划日期：2026-05-16
> 前置版本：v0.5（小白可用 MVP 已发布）
> 规划口径：先稳数据和解释，再优化 UI，最后才考虑策略扩展和 AI。
> 完成口径：截至 2026-05-18，P8.1–P8.7 已完成，系统进入 v0.5 + P8 发布候选完成态。

## 总原则

P8 的目标不是把系统做复杂，而是在不破坏 v0.5 小白体验的前提下，让结果更稳、更好懂、更容易复查。

必须坚持：

- 不碰实盘交易。
- 不改 `v0.5` tag。
- 不做数据库、登录系统、复杂前后端分离。
- 不重写现有 6 因子评分逻辑。
- 不把 AI 放进评分或买卖决策链路。
- 每一阶段都必须保留 v0.5 的双击启动、CLI 验收和截图证据。

## P8.1 数据质量增强

### 目标

把当前“备用数据约 120 条 K 线、覆盖不全普遍出现”的体验，提升到“常用股票至少 250 条 K 线、覆盖不足能明确解释、失败能自动降级”。

### 为什么先做

数据是地基。当前 v0.5 已经能跑，但所有候选都容易出现覆盖不足，小白会自然怀疑结果可信度。P8 如果先做 AI 或多策略，会把复杂度压在不稳的数据上。

### 改动范围

- 数据源评估：先做小型调研和试验，不直接承诺某个库一定接入。
- 多级数据通道：保留现有 `akshare + skill_fallback + cache`，新增数据源必须走可回退链路。
- 缓存增强：记录缓存来源、实际起止日期、K 线条数、更新时间。
- 数据质量输出：CLI JSON 和 UI 都要显示数据来源、覆盖状态和降权原因。

### 验收标准

- `select --universe static --limit 10 --top 5` 成功。
- 10 只 static 股票中至少 8 只有 250+ 条 K 线。
- 覆盖不足比例从当前接近 100% 降到 20% 以下；如果达不到，必须在报告中说明原因。
- 单一数据源失败时，系统仍能自动降级到可用来源。
- `py_compile`、`select`、`backtest-validate`、`report` 全通过。

### 禁止做什么

- 不删除现有 `skill_fallback`。
- 不引入需要付费或 API Key 的数据源作为唯一通道。
- 不改变评分公式和因子权重。
- 不为了追求覆盖率而吞掉数据错误。

## P8.2 策略可解释增强

### 目标

让小白能看懂“为什么这只股票排在前面”“为什么只是观察不是强观察”“为什么风险高或置信度低”。

### 为什么做

v0.5 有评分、风险、置信度，但解释还偏工程化。小白需要的是一句话结论、主要加分项、主要扣分项、需要注意的风险。

### 改动范围

- `strategies/selection.py`：在不改评分的前提下，补充结构化解释字段。
- `app.py`：逐只详情里增加“小白解释”，优先展示中文短句。
- `reports/generator.py`：报告中加入 Top 候选的解释摘要。
- CLI JSON 保留旧字段，新增字段必须向后兼容。

### 验收标准

- 每只候选都有一句话解释。
- 解释至少包含：主要加分因子、主要拖后腿因子、置信度原因、风险提醒。
- 不出现只有英文术语没有中文解释的输出。
- 不改变同一批输入的评分排序。
- CLI 全链路通过。

### 禁止做什么

- 不改评分公式。
- 不新增 AI。
- 不生成投资建议措辞。

## P8.3 UI 体验增强

### 目标

让结果页首屏更像产品，而不是只像数据调试页。小白打开结果后，第一眼就能知道：谁排前面、为什么、风险在哪。

### 为什么做

P7 已经解决“能启动、能看到结果”。P8.3 要解决“结果是否一眼能读懂”。这一步必须建立在 P8.1 数据更稳、P8.2 解释更清楚之后。

### 改动范围

- 选股运行状态：从单纯 spinner 升级为更明确的阶段提示；如果做进度条，必须真实反映处理进度。
- 结果首屏：Tab 外展示 Top 3 候选、整体质量、覆盖风险和下一步提示。
- 风险视觉：高风险红色、中风险橙色、低风险绿色，保持全页面一致。
- 就绪状态：统一首页顶部和“首次使用检查”里的状态判定。

### 验收标准

- 选股运行时，小白能看到当前处于“拉数据/评分/生成报告”等哪一步。
- 结果首屏不用切 Tab 就能看到 Top 3、整体质量和主要风险。
- 截图脚本能捕获首页和结果页，截图路径仍为 `docs/screenshots/`。
- CLI 全链路通过。

### 禁止做什么

- 不换前端框架。
- 不做复杂前后端分离。
- 不把页面做成营销落地页。
- 不牺牲 v0.5 的简单启动路径。

## P8.4 策略管理 ✅ 已完成

### 目标

建立多策略框架，让未来策略可以被注册、选择、验证和比较，但默认策略必须保持不变。

### 为什么放在后面

多策略会增加理解成本。只有当数据质量和解释能力更稳以后，策略选择才有价值，否则小白会被多个名字相似的策略弄晕。

### 已完成内容（真实状态）

- `strategies/registry.py`：元数据注册中心（id/name/description/suitable_scenario/risk_reminder/enabled/entry_function），移除旧的 Skill/脚本执行注册。
- 默认策略 ID 为 `default`，中文名「默认规则策略」。
- CLI：支持 `--strategy default`，省略时行为与之前完全一致。
- Streamlit UI：侧栏增加策略选择器，显示策略名称，help 提示不改变评分公式。
- JSON 顶层新增 `strategy_id` 和 `strategy` 元数据字段（含 id/name/description/suitable_scenario/risk_reminder）。
- 当前只有一套默认策略，未新增第二套策略。
- Markdown 报告生成逻辑未修改，保持兼容；策略元数据不在 Markdown 报告中展示。
- 策略管理只是入口壳，不改变评分、排序、数据链路。

### 验收标准（已完成）

- 不传 `--strategy` 时行为与之前完全一致（仅多了顶层策略元数据）。
- 传 `--strategy default` 时输出一致。
- JSON 顶层包含 `strategy_id` 和 `strategy` 元数据。
- 切换策略框架不影响 `select/backtest-validate/report`。
- py_compile、confirm_p84_registry、confirm_p84_cli、confirm_p84_ui、confirm_p84_docs 全通过。

### 禁止做什么

- 不一次性新增多个未验证策略。
- 不删除或重命名默认策略输出字段。
- 不让策略管理影响 v0.5 用户的默认路径。
- 不把策略名称和版本写入 Markdown 报告作为已完成验收（当前未改报告逻辑）。

## P8.5 AI 辅助解释（未启动，需先通过 P8.5-0 决策评审）

### ⚠️ 进入 P8.5 前必须先做 P8.5-0 决策评审

P8.5 当前处于"规划暂存"状态，**不直接进入实现**。启动前必须先完成 P8.5-0 决策评审，确认以下边界：

1. AI 默认关闭——不需要任何 API Key 才能启动小白 UI。
2. AI 不进评分、排序、风控、交易决策链路——AI 只读取已生成的结果做辅助解释。
3. AI 不输出投资建议措辞——只输出解释和风险提醒。
4. AI 不可用时页面不报错、优雅降级——不影响规则策略的默认路径。
5. 如不能保证以上所有边界，P8.5 暂缓，不启动实现。

**P8.5-0 的产出应该是决策文档，不是代码。**

### 目标

在规则解释已经清楚的基础上，提供可选 AI 辅助解释，帮助小白理解候选股，但不让 AI 参与评分、排序或交易决策。

### 为什么最后做

AI 是解释层，不是地基。只有数据、规则解释、UI 和策略框架都稳定后，AI 才不会变成"看起来聪明但实际遮住问题"的东西。

### 可能的改动范围（需 P8.5-0 确认后才可执行）

- AI 默认关闭。
- 只读取已生成的选股结果、解释字段和风险提示。
- UI 中以折叠区展示"AI 辅助解释"，不可替代规则解释。
- AI 不可用时优雅降级。

### 验收标准（需 P8.5-0 确认后才适用）

- 默认启动不需要任何 API Key。
- AI 关闭时，系统行为和 P8.4 完全一致。
- AI 开启时，只输出解释和风险提醒，不输出投资建议措辞。
- AI 不可用时页面不报错。

### 禁止做什么

- 不让 AI 参与评分、排序、风控等级。
- 不输出投资建议措辞。
- 不把闭源 API Key 作为唯一可用路径。
- 不让 AI 功能影响小白默认启动。
