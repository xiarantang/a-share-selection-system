# 项目来时路复盘（P1-P8）

> 日期：2026-05-18
> 当前基线：v0.5 + P8 发布候选完成态
> 最新基线 commit：`96909eb`

## 1. 总结论

这个项目不是从一开始就是“小白可用产品”。它经历了一个很清楚的演进过程：

> 架构雏形 → 基础可信 → 数据稳定 → 选股 MVP → 批量股票池 → 模型可信度 → 可视化 UI → 小白可用 MVP → P8 发布候选增强

当前最重要的成果不是“功能很多”，而是主路径已经稳定：

> 双击 `start_ui.command` → 选参数 → 开始选股 → 看结果

后续阶段应继续围绕长期可用、结果可信、过程可复盘、小白可理解推进，不应为了炫技引入复杂系统。

## 2. 前置阶段：v0.1 与 P0

### v0.1：架构雏形

初始版本搭出了数据、策略、回测、AI、Agent、模拟交易、报告等模块。这个阶段的价值是让项目有了完整目录结构和发展空间。

但当时它更像工程骨架，不是最终产品形态。后续事实证明，真正重要的不是模块多，而是每条链路能不能稳定、可验收、可解释。

### P0：先去夸大，先保可信

P0 把一些容易造成误解的地方先收住：

- 数据失败时必须明确标记。
- 策略扫描和统计要分清。
- 回测空数据不能假装成功。
- AI 相关能力降为实验性质。
- README 去掉夸大表达。

P0 的意义是立规矩：系统宁可明确失败，也不能输出看似成功但不可信的结果。

## 3. P1：基础可信收口

P1 主要处理退出码、策略统计和文档表述问题。

核心变化：

- 命令失败时返回可信退出码。
- 策略统计与策略能力分开描述。
- 文档表达更克制，不夸大系统能力。

这一阶段让项目从“能跑一些东西”走向“运行结果能被维护者判断是否成功”。

## 4. P2：数据层稳定

P2 重点是让数据链路更抗波动。

核心变化：

- 引入多源 fallback。
- 增加本地缓存。
- 增加 `selfcheck` 命令。
- 修复 fallback JSON 输出、字段映射和超时问题。

这一阶段解决的是地基问题：数据接口会失败，系统必须能识别、降级、缓存和说明。

## 5. P3：选股 MVP

P3 让系统第一次真正具备“批量计算并输出候选结果”的能力。

核心变化：

- 增加 `select` 命令。
- 引入 MA、MACD、RSI 等规则评分。
- 报告接入选股结果。
- 增加覆盖不足提示。
- 全失败时用失败退出码。

P3 的意义是形成第一个可用选股闭环：拉数据 → 算指标 → 评分 → 输出结果 → 生成报告。

## 6. P4：批量选股与股票池

P4 把选股从零散测试推进到可重复的股票池流程。

核心变化：

- 增加 `universe` 模块。
- 支持 sample、hs300、top_amount 等入口。
- 建立 static 股票池。
- 增加股票名称、行业等元数据。
- 报告补充来源说明。
- 输出 latest 文件，方便后续报告和 UI 读取。

P4 的意义是让系统从“测几只股票”变成“按股票池批量运行”。

## 7. P5：模型可信度与验证复盘

P5 是从“有结果”走向“结果能解释、能验证”的关键阶段。

核心变化：

- 升级为 6 因子规则评分。
- 增加 decision、risk_level、confidence。
- 覆盖不足时降权，低置信度时下调展示结论。
- 增加 `validate` 质量摘要。
- 增加 `backtest-validate` 历史窗口复盘。
- Markdown 报告展示验证与复盘摘要。

P5 的意义是：结果不只是一串排名，还包含质量、风险、置信度和复盘线索。

## 8. P6：Streamlit UI 起步

P6 开始把项目从开发者工具转为小白可用界面。

核心变化：

- 增加 Streamlit 本地 UI。
- 页面展示参数、结果、验证摘要和报告。
- 增加 `start_ui.command` 一键启动脚本。
- README 补充小白启动方式。

P6 的意义是让用户不再必须依赖命令行，产品形态开始出现。

## 9. P7：小白可用 MVP 与 v0.5

P7 是产品化打磨阶段，目标是让小白真的能走完流程。

核心变化：

- 首页体验优化。
- 错误提示更友好。
- 数据覆盖信息可视化。
- 真实浏览器验收与截图留证。
- 小白使用指南和排障指南。
- 环境自检页。
- 启动脚本强化。
- 发布清单与变更日志。
- 打出 `v0.5` tag。

P7 的意义是把系统送到“小白可用 MVP”：双击启动、看到页面、运行选股、查看结果、遇到问题有文档可查。

## 10. P8：发布候选增强

P8 在不破坏 v0.5 小白路径的前提下，增强数据、解释、UI、验收和发布边界。

### P8.1 数据质量增强

- 接入 baostock。
- 数据链路稳定为 akshare → baostock → skill_fallback。
- static 前 10 只实测 10/10 为 baostock，约 570 条 K 线。
- 修正覆盖不足误报。
- CLI、报告、UI 三处数据质量展示一致。

意义：把 P7 中普遍覆盖不足的体验，提升为更稳定的数据地基。

### P8.2 策略解释增强

- 新增规则生成的 `explain` 字段。
- JSON、Markdown 报告、Streamlit UI 三处接入解释。
- 解释包含 summary、strengths、weaknesses、risk_note、confidence_note。
- 不引入 AI，不改评分排序。

意义：让小白知道为什么某只股票排在前面、风险在哪里、可靠性如何。

### P8.3 UI 体验增强

- 首页引导统一为“选参数 → 开始选股 → 看结果”。
- 结果首屏增加 Top3 速览。
- 决策、风险、置信度中文化。
- 风险视觉统一。
- 技术指标折叠展示。
- 新增 44 项 UI 自动验收和截图证据。

意义：结果页不再像工程调试页，而是更像一个可读的选股产品。

### P8.4 策略管理

- 建立策略元数据注册表。
- CLI 支持 `--strategy default`。
- UI 增加策略选择器。
- JSON 增加策略元数据。
- 当前仍只有默认规则策略，不新增第二套策略。

意义：为未来扩展留入口，但不让策略扩展影响当前稳定主链路。

### P8.5 AI 边界评审

- 完成 AI 辅助解释边界文档。
- 结论：暂缓实现。
- 明确 AI 不进入评分、排序、风控等级和默认小白路径。

意义：把“看起来高级但边界复杂”的方向先按住，保护系统可信基础。

### P8.6 UI 稳定性与启动验收

- 修复 Streamlit 与 Pandas 废弃 API 小尾巴。
- 更新 requirements 版本边界。
- 真实 UI 冒烟验收。
- 双击启动脚本冒烟验收。
- 更新截图和公开文档。

意义：降低未来依赖升级导致 UI 坏掉的风险。

### P8.7 发布前复盘与发布候选收口

- 增加 `scripts/confirm_release_ready.py`。
- 建立发布前一键验收（P8.7-1 时为 9 项，P9.2 扩展为 10 项，含文档口径一致性检查）。
- 小白手动验收清单收口。
- README、USER_GUIDE、TROUBLESHOOTING、PROJECT_STATE 等公开文档一致性收口。
- 启动入口 fallback 可选状态收口。
- 发布候选结论落章。

意义：P8 不只是做完功能，而是形成可发布、可检查、可交接的完成态。

## 12. 从来时路得到的原则

后续迭代必须继续遵守：

- 不碰实盘交易。
- 不引入 AI/qlib 到评分排序链路。
- 不做数据库、登录系统、复杂前后端分离。
- 不重写评分和排序。
- 不随意改数据链路。
- 不输出投资建议措辞。
- CLI 继续作为底层引擎和验收入口。
- 产品目标继续是小白可视化 UI，而不是命令行工具堆功能。

## 13. 下一阶段方向

P9 不应急着加功能。P9 应围绕四件事：

1. 长期可用：依赖、启动、验收入口更稳。
2. 结果可信：数据来源、质量、解释和验证更清楚。
3. 可复盘：每次结果能追溯参数、数据、报告和验证摘要。
4. 小白可理解：文档、页面提示、排障路径一致。

因此，下一阶段命名为：

> P9：发布版稳定性与复盘可信度增强
