# P8.5-0 AI 辅助解释边界评审决策文档

> 创建日期：2026-05-18
> 阶段：P8.5-0（决策评审，不写代码）
> 前置：P8.4 策略管理已完成

---

## 1. 结论先行

**暂缓实现 P8.5 AI 辅助解释；只有在边界全部可自动验收时，才进入下一步最小设计。**

理由：

1. 当前系统（P8.1-P8.4）已经具备完整的规则解释能力（explain 字段、风险等级、置信度、数据质量说明），小白用户可以通过规则解释理解选股结果。
2. AI 辅助解释属于"锦上添花"而非"地基修复"，收益有限但风险边界复杂。
3. 引入 AI 后的边界守卫成本（禁词检查、默认关闭验证、无 Key 可用验证、隔离验证）不容忽视。
4. 当前阶段更值得投入的方向是 UI/验收稳定性小优化，而非引入一个需要持续守卫的 AI 层。

**如果未来确需推进，必须先完成 P8.5-1 设计文档，且设计文档必须包含可自动验收的边界守卫方案。**

---

## 2. P8.5 的允许定位

如果未来实现 P8.5，其定位**只能是**：

- **解释层增强**：只读取已生成的 `selection_latest.json` 中的 `explain` 字段、`validation` 字段、`risk_note`、`confidence_note`、`reasons`、`risks`、`factor_scores`、`factor_values` 等现有数据。
- **通俗化转换**：将已有的规则解释转换为更通俗的表述、术语解释、风险提醒复述。
- **只读**：不写回任何 JSON 字段，不修改任何评分/排序/风控结果。

允许读取的数据源：
- `selection_latest.json` 的顶层 `top`（候选列表）、`all`（全集列表）和 `explain` 字段
- `strategies/registry.py` 中的策略元数据（`description`、`risk_reminder`、`suitable_scenario`）
- 已有的规则 explain 生成逻辑的输出

---

## 3. P8.5 的硬性禁止

以下禁止项不可协商，任何 P8.5 实现方案违反任何一条即判定为不合格：

### 3.1 不参与评分

AI 不得修改、覆盖、影响任何 `factor_scores`、`total_score`、综合评分计算逻辑。评分公式和因子权重保持 v0.5 + P8.1-P8.4 的现状不变。

### 3.2 不参与排序

AI 不得改变候选股的排名顺序。`top`（候选列表）的排列顺序必须完全由规则评分决定。

### 3.3 不参与风控等级

AI 不得修改、覆盖 `risk_level`（low/medium/high）。风控等级完全由规则逻辑决定。

### 3.4 不参与买卖决策链路

AI 不得修改、覆盖 `decision`（strong_watch/watch/neutral/avoid）。决策标签完全由规则逻辑决定。

### 3.5 不生成股票池

AI 不得新增、删除、替换候选股。候选股列表完全由 `SelectionEngine.select()` 的规则逻辑决定。

### 3.6 不改 data/fetcher.py 数据链路

AI 不得修改数据拉取、缓存、降级逻辑。`data/fetcher.py` 保持 akshare → baostock → skill_fallback 链路不变。

### 3.7 不输出投资建议措辞

AI 输出不得包含以下措辞（正向建议语境）：

- 买入 / 卖出 / 目标价 / 收益预测
- 推荐买入 / 推荐卖出
- 预期收益 / 保证收益
- 建议加仓 / 建议减仓 / 建议清仓
- 止盈位 / 止损位（作为操作指令）
- 看多 / 看空（作为操作建议）
- 任何暗示"应该交易"的表述

### 3.8 不要求 API Key 才能启动小白 UI

系统启动、选股、查看结果、生成报告等所有核心功能的默认路径不得依赖任何 AI API Key。没有 API Key 时系统必须完全可用，且体验与 P8.4 一致。

---

## 4. 默认关闭原则

### 4.1 UI 默认不启用 AI

- Streamlit 页面加载时，AI 辅助解释功能默认关闭。
- 用户必须主动点击或配置才能启用，不允许默认开启。
- AI 开启状态不应持久化到 session_state 以外（不写入配置文件、不写入 JSON）。

### 4.2 CLI 默认不启用 AI

- `main.py` 的所有子命令默认不调用 AI。
- 如需启用，必须通过显式参数（如 `--ai-explain`），且该参数不得有默认值。

### 4.3 没有 API Key 时系统完全可用

- 未配置 API Key 时，所有现有功能正常运行。
- 不得弹出要求输入 API Key 的提示。
- 不得在日志中输出因缺少 API Key 而产生的 warning 影响用户体验。

---

## 5. 可以接受的最小方案（仅设计，不实现）

如果未来 P8.5-1 确认推进，以下是最小可接受方案的设计边界：

### 5.1 展示形式

- UI 中以**可折叠区**展示，标题类似"AI 辅助阅读说明"。
- 折叠区默认收起（`expanded=False`）。
- 折叠区位于规则 explain 下方，不替代规则解释。
- CLI 和 Markdown 报告中默认不输出 AI 解释（除非显式启用）。

### 5.2 输入范围

- 只允许读取已有 JSON 的 Top 候选和规则 explain 字段。
- 不直接读取行情接口、K 线数据、实时价格。
- 不读取外部新闻、公告、研报等未经验证的信息源。

### 5.3 输出范围

- 只能输出：
  - 通俗解释：将 factor_scores / reasons / risks 转为更易懂的表述。
  - 术语解释：如"RSI 是什么""MACD 金叉是什么意思"。
  - 风险提醒复述：将已有的 risk_note 用更通俗的方式重述。
- AI 输出不得写入 `selection_latest.json` 的任何核心字段。
- AI 输出只存在于 UI 渲染层（内存），不持久化。

### 5.4 免责声明

- 每次 AI 解释必须附带免责声明：**"以上为 AI 辅助生成的通俗解释，仅供研究学习，不构成投资建议。选股结果由规则策略计算，AI 不参与评分和决策。"**
- 免责声明不得被折叠、隐藏或弱化。

---

## 6. 不建议做的方案

以下方案明确不建议实现，即使在 P8.5-1 设计阶段也不应纳入考虑：

| 方案 | 不建议原因 |
|------|-----------|
| AI 重新打分 | 违反"不参与评分"硬性禁止 |
| AI 改排序 | 违反"不参与排序"硬性禁止 |
| AI 生成策略 | 系统定位是规则驱动，AI 生成策略会引入不可控风险 |
| AI 给交易动作 | 违反"不参与买卖决策链路"硬性禁止 |
| AI 直接读行情接口并给判断 | 绕过规则评分体系，违反核心定位 |
| AI 替代规则解释 | 规则解释是系统可信基础，AI 只能补充不能替代 |
| AI 预测涨跌 | 违反"不输出投资建议措辞"硬性禁止 |
| AI 选股池 | 违反"不生成股票池"硬性禁止 |
| AI 写入 JSON 核心字段 | 违反只读原则 |
| AI 自动执行交易 | 系统明确不碰实盘交易 |

---

## 7. 进入 P8.5-1 的硬门槛

即使决定推进 P8.5，也必须先满足以下全部门槛才能进入 P8.5-1 设计阶段：

### 7.1 禁词检查验收脚本

- 必须有自动化脚本能检查 AI 输出中的禁词列表（买入、卖出、目标价、收益预测、推荐买入、推荐卖出、预期收益、保证收益等）。
- 禁词检查必须在 CI 或验收流程中可运行。
- 禁词检查失败时必须阻止发布。

### 7.2 默认关闭验收

- 必须能通过自动化方式证明：
  - UI 启动后 AI 功能默认关闭。
  - CLI 不传 AI 参数时不调用 AI。
  - 不存在任何隐式启用 AI 的路径。

### 7.3 无 API Key 可启动验收

- 必须能证明：在没有配置任何 AI API Key 的环境下：
  - `select` 正常运行且结果与 P8.4 一致。
  - `backtest-validate` 正常运行。
  - `report` 正常运行。
  - Streamlit UI 正常启动且所有功能可用。

### 7.4 默认路径不调用 AI 验收

- 必须能证明 `app.py` / `main.py` 的默认执行路径中不存在 AI 模块的 import 调用或初始化。
- AI 模块必须采用延迟导入（lazy import），仅在用户主动启用时才加载。

### 7.5 输出隔离验收

- 必须能证明 AI 输出只进入可折叠解释区，不进入 JSON 核心评分字段。
- `selection_latest.json` 不得因 AI 功能而新增任何字段。
- AI 输出不得出现在 Markdown 报告中（除非显式启用且报告生成时用户确认）。

## 8. 最终建议

**下一步不是实现 P8.5，而是：**

1. **首选**：暂缓 P8.5，转向更实用的小白体验优化和验收稳定性优化。例如：
   - 补充更多边界情况的错误提示。
   - 优化选股耗时较长时的用户体验。
   - 补充更多验收脚本的覆盖。
   - 更新截图和用户文档中的细节。

2. **如确需推进**：先做 P8.5-1 设计文档，设计文档必须包含：
   - 可自动验收的边界守卫方案（禁词、默认关闭、无 Key 可用、路径隔离、输出隔离）。
   - 最小方案的详细设计（折叠区位置、输入/输出格式、免责声明样式）。
   - 验收脚本的设计（不是实现）。
   - 回滚方案（如何在不破坏 P8.4 状态的情况下移除 AI 功能）。

3. **无论如何**：P8.5 不得在以下条件未满足时进入实现阶段：
   - 本决策文档的所有硬性禁止已被验收脚本覆盖。
   - 至少一名 reviewer 确认边界守卫方案可行。
   - 项目状态文件已更新为"P8.5-0 完成，P8.5 实现暂缓"。
